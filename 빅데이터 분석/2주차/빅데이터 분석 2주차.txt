Hadoop HDFS ( Hadoop distributed file system)
- 분산 데이터 저장소 ( 네트워크 기반 파일 시스템 )
- 빅데이터를 하둡 클러스터 컴퓨터들에 분산 저장하는 시스템
- 클러스터의 하드드라이브를 하나의 거대한 파일 시스템처럼 사용
- 데이터 여분 복사본 생성

Hadoop Map Reduce
- 분산 처리를 위한 필수 처리 방식
- Mapper와 reducer로 구성됨 ( 2개의 구분된 스크립트 or 함수 )
- Mapper : 클러스터에 분산된 데이터를 효율적으로 동시에 변형 ( transform )
              최종적으로 원하는 결과 고려해 데이터를 변형하는 단계임
- Reducer: 변형된 데이터 집계 ( aggregate )
- Shuffling: 데이터 분류 ( 직접 코딩 X )

Hadoop YARN ( Yet Another Resource Negotiator )
- Hadoop의 클러스터 자원 관리 시스템
- 어떤 노드(컴퓨터)가 작업을 언제 실행하고, 추가 작업을 할 수 있는지 없는지 결정

Apache Pig - language
- MapReduce 위 고수준 스크립팅 언어 
- SQL과 유사한 Pig Latin 자체 스크립트 언어로 처리
- Pig Latin을 해석해 MapReduce로 변환해 실행

Hive - language
- Hadoop에서 동작하는 Data warehouse ( 데이터 관리 시스템 ) -> 저장처리 + 언어
- 데이터 요약, 쿼리 및 분석 기능 제공
- ODBC ( Open Database Connectivity )를 통해 데이터베이스 접속
- HDFS가 정리안된 책더미라면 Hive는 장르별, 저자별로 정리된 도서관
- HiveQL 쿼리 -> SQL과 유사

Apache Ambari
- 클러스터 관리와 모니터링 간소화
- 사용하기 쉬운 UI와 REST API를 통해 구현
- 클러스터 모니터링, 구성 변경 때 사용

Mesos ( 중요 X , 안배움 )
- Yarn과 비슷한 용도로 리소스 관리 기능 수행
- 클라우드 인프라스트럭쳐 및 컴퓨팅 엔진의 다양한 자원을 통합적으로 관리
- Yarn은 Hadoop전용이지만 Mesos는 범용 cloud O.S이며 전체 서버를 관리함
- Hadoop의 세부적인 스케쥴링은 Yarn이 나음 그러나 Mesos는 Hadoop 외 다른 작업과도
  자원관리 가능

Apache Spark
- Map Reduce 아이디어 계승 -> 실행하는 엔진을 새롭게 설계
- 클러스터 환경에서 데이터를 병렬로 처리하는 라이브러리 집합
- 데이터 신속하고 효율적, 안정적으로 처리
- 파이썬, 자바, 스칼라, R 지원
- SQL, 스트리밍, 머신러닝 등 넓은 범위 라이브러리 제공
- Hadoop의 단점인 처리속도를 높였지만 메모리 점유가 높다는 단점 존재
- Hive 기능 거의 완벽하게 대체

TEZ ( 실행 엔진 )
- Hive + TEZ가 주로 쓰임
- 방향성 비순환 그래프 ( DAG: Directed Acyclic Graph ) 사용 -> 쿼리 실행 효율적

※ Hive + TEZ    /   Spark  
   안정적              메모리 점유가 높음 ( 관리 까다로움 )
   SQL( HiveQL )     SQL, ML, Streaming
                         속도가 조금 더 빠름
   중요 Data 통계    데이터 분석, AI

HBASE
- HDFS 기반 NoSQL 데이터베이스
- 대규모 클러스터에서도 빠른 성능

Apache STORM
- 분산형 스트림 프로세싱 연산 프레임워크
- 스트리밍 데이터 실시간 처리
- 실시간 분석, 머신러닝, 운영 모니터링 용도

Oozie
- 클러스터 작업 스케쥴링 ( 워크플로우 관리 )
- 작업 순서 제어, 의존성 관리 ex) 데이터 오염되면 3번 중단,
  조건부 실행 ex) 새 데이터 들어오면 1번부터 시작 

ZooKeeper
- 분산 코디네이션
- 클러스터 전체 노드를 모니터링 -> Master가 죽었을 때 새 Master 선정 ( 동기화 )

Sqoop, Flume, Kafka
- Sqoop: 데이터베이스와 Hadoop 잇는 연결 장치
- Flume: 실시간 데이터를 안정적으로 클러스터에 불러옴
- Kafka: PC나 서버 클러스터에서 모든 종류 데이터 수집

블록 사이즈
- HDFS의 기본 블록 사이즈는 일반적인 파일 시스템 또는 데이터베이스보다 큰 128MB
- Disk seek time 감소: 블록 사이즈 이용해 탐색 시간 줄임
- 네임노드의 메타데이터 크기 감소: 블록 사이즈 클 경우 메타 데이터 크기 감소
- 클라이언트와 네임노드 간 통신 감소: 블록이 커짐에 따라 블록 수 줄어들면서
                                                 파일 접근 횟수 감소

Name node ( HDFS의 매니저 )
- 메타데이터 ( 데이터의 데이터 ) 관리: 파일시스템 유지 위한 메타데이터 관리
                        메타데이터는 파일명, 디렉토리, 크기, 권한, 블록 위치 정보 등으로 구성
- 데이터노드 모니터링: 데이터노드는 네임노드에 주기적으로 heartbeat( 동작 중 알림 )와 
                              block report ( 블록 위치 정보 ) 전송
- 블록 관리: 데이터노드 장애 발생 시, 블록을 새로운 데이터노드로 복제
                데이터노드 용량 부족한 경우 블록 이동
- 클라이언트 요청 접수: 클라이언트가 HDFS에 접근하기 위해 네임노드에 접속해
                               권한 확인 절차 필요

Data node ( block 저장 )
- 클라이언트가 HDFS에 저장하는 파일을 로컬디스크에 유지
- 블록의 목록을 주기적으로 네임노드에 보고
- Name node가 Data node의 블록 구조 관리하지만 데이터의 읽기와 쓰기는 클라이언트와
  Data node에 의해 수행

Name node의 복원 ( Summary: 메타데이터 백업 )
- 메타데이터 백업: 로그를 로컬 디스크와 NFS에 저장 ( 백업 데이터 작성 소요시간 중 일부
                                                                    데이터 소실 가능성 있음 )
- 세컨드 네임노드: 주 네임노드 로그 복사본 유지 ( edit log로 주기적인 메타데이터 
                                                               업데이트 )
- HDFS federation: 서브디렉터리마다 분리된 네임노드 지정 ( 구역 별 매니저 설정 )
- edit log를 HDFS가 아닌 다른 안정한 저장소 ( active name node )에 저장
- Zookeeper는 active name node 파악하고, 하나의 네임노드만 사용하게 통제

Mapping
- 데이터 변경, 로직수행
- 데이터로부터 필요한 정보 추출 후 이해할 수 있는 구조로 변경

Reducing
- Mapping의 결과 집계

Handling failure
- Application master ( 작업 매니저 ) 가 문제 발견한 경우 -> 작업 재시작, 다른 노드 컨테
                                                                            이너가 해당 작업 재시도
- Application master 기능이 정상 동작 X -> YARN resource manager가 가능한 노드에서           
                                                        작업 재시작
- YARN resource manager에 문제 발생 -> Zookeeper 이용해 자동으로 백업 
                                                      resource manager 지정
